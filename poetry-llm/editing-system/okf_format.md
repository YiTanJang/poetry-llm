# OKF 편집 규칙

## Frontmatter

모든 `.md` 파일의 맨 위에 YAML frontmatter를 포함한다.

```yaml
---
type: Design
title: 파일 제목
description: 파일의 현재 상태와 핵심 결정을 알 수 있는 요약 (TL;DR 역할)
tags: [태그1, 태그2]
timestamp: 2026-06-28T00:00:00Z
---
```

`description` 필드는 에이전트가 파일 전체를 읽지 않고도 상태를 파악할 수 있도록 **구체적인 밀도 높은 요약(TL;DR)**으로 작성한다.
예: `description: 모델 선택 (진행중) - Gemma 27B 우선 검토, LoRA 대신 풀 파인튜닝으로 가닥 잡음`

### `type` 값

| type | 용도 |
|------|------|
| `Bundle Index` | 번들 루트 또는 섹션 index.md |
| `Vision` | 프로젝트가 왜 존재하는가 |
| `Goals` | 구체적 목표와 비목표 |
| `Roadmap` | 단계별 진행 계획 |
| `Design` | 설계 결정 문서 |
| `Data Source` | 데이터 소스 기술 |
| `Playbook` | 절차·실행 가이드 |
| `Reference` | 관련 연구, 참고 자료 |
| `Open Questions` | 미답 질문 목록 |
| `Bundle Log` | 변경 이력 |

## 마커 시스템

### 열린 목록 — `> 탐색중`

섹션 첫 줄에 이 마커가 있으면 항목을 자유롭게 추가할 수 있다.
- **상태 전이 (Definition of Done)**: 에이전트는 스스로 `> 탐색중` 마커를 제거할 수 없다. 해당 섹션 내의 모든 `[TODO]`가 해결되고 사용자의 명시적 승인이 있을 때만 마커를 제거하여 확정 상태로 전환한다.
- 마커가 없는 섹션 = **확정 (Accepted)**. 불변(Immutable) 원칙에 따라 기존 내용을 덮어쓰거나 변경하는 것을 금지한다.

### 수정 제안 블록 — `> 수정 제안:`

확정된(탐색중 마커가 없는) 문서의 내용을 수정해야 할 경우, 원본을 직접 수정하지 말고 해당 내용 하단에 아래 블록을 추가한다.
```markdown
> 수정 제안: 새로운 모델 벤치마크 결과에 따라 기존 내용을 X에서 Y로 변경 제안
```
이후 사용자가 리뷰하여 승인하면 원본을 업데이트한다.

### 미결 사항 및 결정 사항 마커

각 파일 하단의 `## 미결 사항`에서 항목 앞에 붙인다.

| 마커 | 의미 |
|------|------|
| `[TODO]` | 지금 당장 작업 가능 |
| `[User Review]`| **사용자 컨펌 필수.** 에이전트는 절대 스스로 해결하려 시도하지 않고 대기/질문 |
| `[Ph1]` | Phase 1 이후 결정 |
| `[Ph2]` | Phase 2 이후 결정 |

**미결 사항이 해결된 경우 (Decision Record)**:
해결된 미결 사항은 단순히 삭제하지 않고 `## 결정 사항 (Decisions)` 섹션으로 이동시킨 뒤, 아래 형식으로 남겨 맥락(History)을 보존한다.
```markdown
- [해결됨: 2026-06-28] 프롬프트 분리 처리 결정 (이유: 하나의 거대 프롬프트보다 CoT와 생성 단계를 나누는 것이 품질이 높음)
```

### 가설 블록

```markdown
> 가설: 검증되지 않은 주장을 여기에 쓴다.
```

검증 완료 시 블록 제거, 일반 텍스트로 승격.

### Citations

```markdown
# Citations

- 저자 (연도) "논문 제목"
```

아직 읽지 않은 논문은 기재하지 않는다.

## 에이전트 인수인계 (Handoff) 로그

작업 세션이 끝나면 `poetry-llm/log.md`에 아래 포맷으로 Handoff 로그를 남겨 다음 작업자(또는 다음 에이전트 세션)가 컨텍스트를 즉시 로드할 수 있게 한다.

```markdown
- **[YYYY-MM-DD] 작업 요약**: (예: 토큰화 방식 결정 및 관련 문서 업데이트)
  - **수정 파일**: `preprocessing/tokenization.md`
  - **핵심 결정**: BPE 대비 형태소 분석기 병행 사용 확정
  - **다음 단계 (Next Steps)**: `evaluation/` 폴더 내에 형태소 보존 평가 지표 구체화 필요
```

## 다른 섹션 참조

다른 섹션 파일을 참조할 때는 **링크만** 사용. 파일명 변경 시에도 맥락을 유지할 수 있도록 개념을 함께 명시한다.

```markdown
→ 모델 선택 전략(`model/model_selection.md`) 참조
```

## 파일 명명

- 소문자 + 언더스코어: `model_selection.md`, `token_budget.md`
- 영어 파일명 (내용은 한국어 무방)

## 커밋 메시지

```
[섹션] 간결한 설명

예:
[data] 한국 현대시 수집 전략 구체화
[model] CPT 스케일링 법칙 인용 추가
[research] Q29-Q35 파이프라인 블로커 추가
```
