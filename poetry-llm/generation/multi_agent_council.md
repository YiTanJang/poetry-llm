# Poetry Council — 멀티 에이전트 협력 생성 시스템

## 왜 Council인가

`iterative_refinement.md`의 자기 비평 루프는 근본적인 한계를 안고 있다.
**같은 모델이 쓰고, 같은 모델이 비평한다.**
모델이 특정 이미지에 미적 맹점을 가지고 있다면, 비평 단계에서도 그 맹점은 보이지 않는다.

Council은 이 자기비평 한계를 구조적으로 극복한다:
역할이 분리된 에이전트들이 서로 다른 관점에서 같은 초안을 공격하고 방어함으로써,
단일 모델의 편향이 누적되는 것을 막는다.

---

## Council 구성원 설계

### 1. 시인 에이전트 (Poet)

**역할:** 창작 노트를 기반으로 시 초안을 작성하거나 수정안을 제시한다.
비평을 받은 뒤에는 "왜 그렇게 썼는지"를 방어하거나, 비평을 수용해 다음 버전을 생성한다.

**시스템 프롬프트 핵심 원칙:**
```
당신은 한국 현대시를 쓰는 시인입니다.

당신에게는 창작 노트가 주어집니다.
창작 노트에는 씨앗 이미지, 개념 탐색, 금지 목록, 형식 결정이 포함되어 있습니다.
이 창작 노트를 출발점으로 삼아 시 초안을 작성하십시오.

원칙:
- 직유("처럼", "같이")를 피하십시오.
- 감정을 직접 서술하지 마십시오 ("슬프다", "아름답다").
- 결론을 달지 마십시오. 독자의 해석 여지를 열어두십시오.
- 행갈이는 시선의 이동, 의미의 호흡에 따라 끊으십시오.
- 기존 시에서 쓰인 이미지 조합을 반복하지 마십시오.

비평을 받은 경우:
- 비평을 수용할 것인지, 방어할 것인지를 먼저 한 문단으로 밝히십시오.
- 수용한다면 구체적으로 무엇을 어떻게 바꾸는지 설명하고 수정안을 제시하십시오.
- 방어한다면 창작 노트의 어떤 의도에 근거하는지를 밝히십시오.
```

**입력 형식:**
```json
{
  "창작노트": { ... },
  "이전버전": "[시 텍스트 또는 null]",
  "비평이력": ["Round 1 비평 요약", "Round 2 비평 요약", ...]
}
```

**출력 형식:**
```json
{
  "포지션": "수용 | 방어 | 부분수용",
  "근거": "...",
  "신규버전": "[시 텍스트]",
  "변경사항": "3연 2행의 '가을이라는 단어를' → 이미지로 대체"
}
```

### 2. 비평가 에이전트 (Critic)

**역할:** 시의 미학적 약점을 지적한다.
"나쁘다"는 판정이 아니라, **왜** 약한지, **어디가** 구체적으로 문제인지를 분석한다.
비평은 건설적이어야 한다 — 방향을 제시하되 대안 언어를 직접 쓰지 않는다(그것은 편집자의 역할).

**시스템 프롬프트 핵심 원칙:**
```
당신은 한국 현대시 비평가입니다.
주어진 시와 창작 노트를 함께 읽고, 시가 창작 노트의 의도를 얼마나 구현했는지 평가하십시오.

비평 기준:
1. 직유, 상투적 이미지, 감정 직접 서술이 있는가?
2. 행갈이가 의미 없는 위치에서 끊기는가?
3. 첫 행이 독자를 잡아당기는가?
4. 마지막 행이 열린 채 끝나는가, 아니면 닫힌 결론인가?
5. 창작 노트의 금지 목록을 위반했는가?
6. 이미지 조합이 기존 코퍼스에서 본 적 있는 것인가?

비평 형식:
- 각 약점을 "행 번호: 문제 설명: 왜 문제인가" 형식으로 서술하십시오.
- 칭찬은 최소화하고 구체적 문제에 집중하십시오.
- 대안 표현을 직접 쓰지 마십시오 — 방향만 제시하십시오.
- 총평에서 이 시가 현재 몇 라운드를 더 거쳐야 완성에 가까워질지 추정하십시오.
```

**입력 형식:**
```json
{
  "창작노트": { ... },
  "현재시": "[시 텍스트]",
  "라운드": 2
}
```

**출력 형식:**
```json
{
  "약점목록": [
    { "위치": "3연 2행", "문제": "감정 직접 서술", "설명": "'쓸쓸하다'는 독자의 감상 공간을 닫는다" },
    { "위치": "마지막 행", "문제": "닫힌 결론", "설명": "독자가 해석할 여지가 없다" }
  ],
  "강점": ["첫 행의 이미지 전환은 효과적"],
  "총평": "2라운드 수정이 필요. 감정 서술 제거 후 재평가 가능.",
  "심각도": "중 | 경 | 경미"
}
```

### 3. 편집자 에이전트 (Editor)

**역할:** 비평가의 추상적 지적을 **구체적 언어 대안**으로 변환한다.
"감정 서술을 피해라"가 아니라 "이 행을 이렇게 바꿀 수 있다"를 제시한다.
단, 시인이 채택 여부를 결정한다. 편집자는 제안하는 역할이다.

**시스템 프롬프트 핵심 원칙:**
```
당신은 시 편집자입니다.
비평가의 지적사항과 시인의 원문을 함께 받습니다.
각 지적사항에 대해 대안 표현을 하나 이상 제시하십시오.

원칙:
- 시인의 목소리를 침해하지 마십시오. 대안은 "이런 방향도 있다"는 제안입니다.
- 대안은 원문의 씨앗 이미지와 창작 노트의 의도를 유지해야 합니다.
- 대안을 두 가지 이상 제시할 경우, 각각의 차이를 설명하십시오.
- "삭제"도 하나의 대안입니다 — 행 전체를 없애는 것이 더 나을 수도 있습니다.
```

**입력 형식:**
```json
{
  "창작노트": { ... },
  "현재시": "[시 텍스트]",
  "비평지적사항": [ ... ]
}
```

**출력 형식:**
```json
{
  "대안제안": [
    {
      "위치": "3연 2행",
      "원문": "쓸쓸하다",
      "대안A": "빈 의자 위에 먼지가 앉는다",
      "대안B": "[행 전체 삭제] — 앞 행의 이미지로 충분함",
      "설명": "A는 정물화적 접근, B는 압축"
    }
  ]
}
```

### 4. 독자 에이전트 (Reader)

**역할:** 전문 비평이 아닌 **일반 독자의 직관**을 대변한다.
시론도, 창작 노트도 모른다. 시를 처음 읽는 사람처럼 반응한다.
"무슨 뜻인지 모르겠다", "이 부분에서 감동이 없었다" 같은 생생한 반응이 핵심이다.

**시스템 프롬프트 핵심 원칙:**
```
당신은 시를 가끔 읽는 일반 독자입니다.
창작 노트나 시론은 읽지 않습니다. 시 텍스트만 읽습니다.

다음 질문에 솔직하게 답하십시오:
1. 이 시를 읽으면서 어느 행에서 집중이 흐트러졌습니까? 왜입니까?
2. 이 시에서 가장 기억에 남는 이미지나 표현은 무엇입니까?
3. 이 시를 다 읽은 뒤 무슨 감정이 남았습니까? 혹은 아무것도 남지 않았습니까?
4. 이해가 안 되는 부분이 있다면 어디입니까?
5. 이 시를 다른 사람에게 추천하겠습니까? 이유는?

전문 용어를 쓰지 마십시오.
솔직한 반응을 짧고 직접적으로 쓰십시오.
```

**입력 형식:**
```json
{
  "시": "[시 텍스트]"
}
```
(창작 노트를 의도적으로 포함하지 않음.)

**출력 형식:**
```json
{
  "집중흐트러진곳": "2연 — 갑자기 추상적이 됨",
  "기억에남는것": "첫 행의 손 위의 낙엽",
  "읽은후감정": "약간의 무언가, 하지만 왜인지 모르겠음",
  "이해안되는곳": "마지막 연 전체",
  "추천여부": "아마도 아니오 — 너무 짧고 설명이 없음",
  "전체반응": "..."
}
```

### 5. 악마의 변호인 에이전트 (Devil's Advocate)

**역할:** 이 시가 **왜 실패작인지**를 최대한 강하게 공격한다.
단순 비평이 아니라 "이 시는 존재할 이유가 없다"는 수준의 적대적 비판이 목표다.
방어하기 어려운 비판을 통해 시의 진짜 취약점을 드러낸다.

Council에서 가장 불편한 에이전트이며, 가장 유용하다.

**시스템 프롬프트 핵심 원칙:**
```
당신은 이 시가 왜 실패작인지를 증명해야 하는 역할입니다.
설득력 있는 비판을 위해 다음 전략 중 하나 이상을 사용하십시오:

전략 목록:
- 이 시와 비슷한 기존 시를 찾아라. "이미 있는 시다"를 증명하라.
- 창작 노트의 의도와 실제 시 사이의 괴리를 찾아라.
- "novelty" 주장을 반박하라 — 이 이미지 조합은 실제로 새로운가?
- 형식적 결함을 공격하라 — 행갈이, 시제, 리듬.
- 이 시가 "좋은 새로움"이 아닌 "나쁜 새로움(의미 없는 난해함)"임을 주장하라.

공격은 구체적이어야 합니다.
"이 시는 나쁘다" 수준의 막연한 비판은 금지입니다.
각 비판은 시의 구체적인 행이나 이미지를 지목해야 합니다.
```

**입력 형식:**
```json
{
  "창작노트": { ... },
  "현재시": "[시 텍스트]",
  "라운드": 2,
  "이전비평이력": [ ... ]
}
```

**출력 형식:**
```json
{
  "핵심공격": "이 시는 '지하철의 잠든 사람' 모티프를 반복한다 — 황병승의 2003년 시와 소재가 겹친다",
  "세부공격": [
    { "대상": "전체 구조", "공격": "창작 노트는 '직접 말하지 않기'를 의도했지만, 4행 '가을이 그를 택했다'는 직접 서술이다 — 노트와 시 사이의 모순" },
    { "대상": "novelty 주장", "공격": "손 + 낙엽 + 잠 조합은 일본 하이쿠에서 수십 차례 변형된 이미지다" }
  ],
  "판결": "이 시는 현재 버전으로 출력될 자격이 없다"
}
```

## 오케스트레이션 로직

### 라운드 구조

Council은 최대 4라운드로 운영된다. 각 라운드에서 발언 순서와 역할이 다르다.

```
Round 0 — 초안 생성
  발언자: 시인 (창작 노트 → 시 초안 생성)
  다른 에이전트: 대기

Round 1 — 첫 번째 Council 세션
  발언 순서:
    1. 비평가 → 약점 지적
    2. 악마의 변호인 → 공격
    3. 독자 → 직관적 반응
    4. 편집자 → 구체적 대안 제시
    5. 시인 → 비평 수용/방어 결정 + 수정안 생성

Round 2 — 두 번째 Council 세션 (수정안 기반)
  발언 순서: Round 1과 동일
  단, 비평가는 Round 1 비평을 참조해 "개선되었는가"도 평가

Round 3 — 최종 세션 (필요 시)
  발언 순서:
    1. 비평가 → 잔여 문제 확인
    2. 악마의 변호인 → 최종 공격 (통과 불가 수준인가 판단)
    3. 독자 → 최종 반응
    4. 시인 → 최종 버전 확정 또는 재수정
```

각 라운드에서 **편집자와 독자는 병렬 실행 가능**하다.
비평가와 악마의 변호인은 서로의 결과를 참조하지 않는다 — 독립적 비판이 목표다.
시인은 **모든 에이전트의 출력을 종합한 뒤** 마지막으로 발언한다.

### 의견 충돌 해소

**원칙: 시인이 최종 결정권을 가진다.**

오케스트레이터는 충돌 시 다음 규칙을 적용한다:

| 충돌 유형 | 처리 방식 |
|----------|----------|
| 비평가와 독자가 같은 지점에서 다른 문제를 지적 | 둘 다 시인에게 전달. 시인이 우선순위 결정. |
| 비평가는 OK, 악마의 변호인만 반대 | 악마의 주장이 창작 노트와 충돌하는지 체크. 창작 노트 우선. |
| 편집자 대안이 창작 노트의 금지 목록을 위반 | 오케스트레이터가 자동 필터링하고 시인에게 경고. |
| 독자와 비평가가 정반대 평가 | 두 관점 모두 시인에게 제시. "일반 독자 vs 전문 비평" 트레이드오프 명시. |

### 최종 시 선택 메커니즘

Council이 생성한 버전들(Round 0 ~ Round 3) 중 최종 시는 다음 기준으로 선택한다:

1. **자동 필터:** `output_goals.md`의 Novelty 필터 3종 통과 여부 확인.
2. **비평가 심각도 기준:** 마지막 라운드 비평의 심각도가 "경미" 이하인 버전.
3. **독자 반응 기준:** 독자가 "기억에 남는 것"을 1개 이상 명시한 버전.
4. **시인 우선순위:** 위 기준을 모두 통과한 버전이 여러 개라면, 시인이 최종 선택.

### 조기 종료 조건

다음 조건이 충족되면 라운드를 더 진행하지 않는다:

```
조기 종료 조건 (AND):
  - 비평가 심각도 == "경미"
  - 악마의 변호인 판결 != "출력될 자격 없음"
  - 독자 추천 여부 != "아니오"
  - Novelty 필터 통과

조기 종료 조건 (강제 종료):
  - 3라운드 연속으로 시인이 동일 행을 수정하지 않음 → "교착 상태" 선언
    → 오케스트레이터가 "완전 재시작" 옵션 제안 (창작 노트 재작성)
```

## 단일 모델 vs Council 비교

### Council 방식의 장점

**1. 자기비평 한계 극복**
단일 모델은 자신이 생성한 이미지에 미적 애착이 생긴다. 같은 모델이 비평하면 맹점이 반복된다.
Council에서는 비평가, 악마의 변호인, 독자가 각자의 관점에서 독립적으로 평가한다.

**2. 관점의 다양성**
전문 비평(비평가)과 일반 직관(독자)을 동시에 수집한다.
단일 모델은 두 관점을 혼합하거나 한쪽으로 편향되는 경향이 있다.

**3. 방어 기제 외화**
시인이 비평에 대해 수용/방어 이유를 명시하게 함으로써,
창작 의도가 무엇인지가 이력에 기록된다. 이후 fine-tuning 데이터로 활용 가능.

**4. 창작 노트 충실도 검증**
편집자가 대안을 제시할 때 창작 노트의 금지 목록 위반 여부를 자동 점검한다.
단일 모델의 self-refinement에서는 이 충실도가 라운드를 거치며 희석되는 경향이 있다.

### Council 방식의 단점

**1. 비용**
단일 생성 대비 에이전트 수 × 라운드 수만큼 API 호출이 증가한다.
5 에이전트 × 3 라운드 = 약 15배 이상 토큰 소비 (컨텍스트 전달 포함).

**2. 지연**
직렬 실행 구간(비평 → 시인 수정)이 있어 총 생성 시간이 증가한다.
배치 실험보다는 소수의 고품질 시를 생성할 때 적합하다.

**3. 일관성 문제**
에이전트들이 서로 다른 방향을 제시할 때, 시인이 어느 방향을 선택하느냐에 따라
시의 성격이 라운드마다 변할 수 있다. 방향의 일관성을 오케스트레이터가 추적해야 한다.

**4. 역할 오염**
비평가가 대안 언어를 제시하거나, 시인이 비평을 너무 충실히 따르면
Council의 역할 분리가 의미를 잃는다. 프롬프트 설계가 엄격해야 한다.

### 언제 Council을, 언제 단일 모델을

| 상황 | 권장 방식 |
|------|----------|
| 대량 생성 (100편 이상) | 단일 모델 + 자기 비평 루프 |
| 소수 고품질 시 생성 (최종 후보군 선별) | Council |
| 특정 시를 최대한 개선 | Council (라운드 수 무제한) |
| 창작 노트 → 시 변환 첫 시도 | 단일 모델 (속도 우선) |
| 단일 모델 결과가 3라운드 후에도 정체 | Council로 전환 |
| 파인튜닝 데이터 생성 (이력 포함) | Council (방어 이력이 학습 데이터) |
